# Operations Guide

## Cron Setup

Five active cron jobs (all times are local machine time):

```crontab
# Main trading loop -- every minute
* * * * * cd /Users/nayslayer/rivalclaw && /Users/nayslayer/rivalclaw/venv/bin/python run.py --run >> /Users/nayslayer/rivalclaw/rivalclaw.log 2>&1

# Self-tuner + hourly report + changelog -- top of every hour
0 * * * * /Users/nayslayer/rivalclaw/venv/bin/python run.py --tune >> /Users/nayslayer/rivalclaw/rivalclaw.log 2>&1

# Paper monitor (Telegram reports) -- every 5 minutes
*/5 * * * * /Users/nayslayer/rivalclaw/venv/bin/python /Users/nayslayer/rivalclaw/paper_monitor.py >> /Users/nayslayer/rivalclaw/paper_monitor.log 2>&1

# Daily report generation + git push -- 04:00
0 4 * * * bash /Users/nayslayer/rivalclaw/daily-update.sh >> /Users/nayslayer/rivalclaw/daily-update.log 2>&1

# Hourly git sync (via commander) -- :27 past each hour
27 * * * * cd /Users/nayslayer/openclaw-audit && /usr/bin/python3 commander/commander.py --claw rivalclaw >> commander/logs/cron.log 2>&1
```

### What Each Job Does

**run.py --run**: Executes one full cycle: fetch markets from Kalshi (+ Polymarket
if enabled), run 8-strategy brain analysis, risk engine filtering, execute paper
trades, route live/shadow orders, check stops, resolve settled markets, daily
snapshot.

**run.py --tune**: Runs the self-tuner (realized vol, strategy threshold, slippage
calibration), generates the hourly report, appends to CHANGELOG.md, sends the
hourly Telegram report.

**paper_monitor.py**: Queries Kalshi API for live account data (balance, settlements,
positions) and paper_trades DB for paper stats. Sends a combined live+paper report
to Telegram.

**daily-update.sh**: Queries DB for daily stats, generates a markdown report in
`daily/YYYY-MM-DD.md`, runs the Strategy Lab research cycle, commits and pushes
to GitHub.

**commander**: External process (in openclaw-audit) that commits log/state changes
and pushes to GitHub + GitLab every hour at :27.

## Process Locking

**File:** `run.py` -- `_acquire_lock()`

Uses `fcntl.flock()` with `LOCK_EX | LOCK_NB` on `.rivalclaw_run.lock`. If another
instance is already running, the new one prints "Another run is already in progress"
and exits with code 0 (not an error).

The lock is released in a `finally` block after `run_loop()` completes. The lock
file is never deleted -- only the flock is released.

## Monitoring

### Paper Monitor (paper_monitor.py)

Runs every 5 minutes via cron. Reports two sections:

**LIVE section**: Kalshi API data only.
- Account balance and portfolio value (from `kalshi_executor.get_balance()`).
- Settlement PnL: queries Kalshi fills API, computes W/L/PnL per series.
- Open positions: queries Kalshi positions API.
- Resting orders and live order counts from DB.

**PAPER section**: From paper_trades table.
- Closed trades since paper cutoff date.
- PnL, win rate, ROI by strategy.
- Open positions count and unrealized PnL.

Messages are sent via Telegram Bot API to `CHAT_ID`.

### Hourly Reports

Generated by `hourly_report.py` (triggered by `run.py --tune`). Written to
`daily/hourly-latest.md` and sent via Telegram by `notify.send_hourly_report()`.

Includes: balance, PnL, win rate, regime classification, top strategies,
recent trades, tuner adjustments.

## Telegram Integration

**Bot token:** Set via `TELEGRAM_BOT_TOKEN` env var (or hardcoded in paper_monitor).
**Chat ID:** Set via `TELEGRAM_CHAT_ID` env var.

**What gets reported:**

| Source | Frequency | Content |
|--------|-----------|---------|
| paper_monitor.py | Every 5 min | Live balance, settlements, paper PnL |
| notify.py (hourly) | Every hour | Full hourly performance summary |
| execution_router.py | Per trade | Order submitted/filled/rejected alerts |
| kalshi_executor.py | On event | Slippage warnings (>500 bps), rate limit hits |

Messages are chunked to 4000 chars (Telegram limit is 4096). On Markdown parse
failure, retries without parse_mode.

## Log Files

| File | Source | Content |
|------|--------|---------|
| `rivalclaw.log` | run.py --run, --tune | Main cycle output: fetches, decisions, trades, errors |
| `paper_monitor.log` | paper_monitor.py | Monitor output and Telegram send confirmations |
| `risk_debug.log` | risk_engine.py | Every blocked/sized decision with full reasoning |
| `daily-update.log` | daily-update.sh | Daily report generation and git push output |
| `logs/dispatcher.log` | commander | Hourly sync commit/push output |

`rivalclaw.log` is the primary diagnostic file. Each cycle prints:
- Market fetch counts (Polymarket + Kalshi).
- Wallet state (balance, open positions, win rate, spot prices).
- Brain output count, risk engine output, blocked count, regime.
- Each executed or rejected trade with reasoning.
- Stop closures with PnL.
- Cycle timing breakdown (fetch/analyze/wallet/total ms).

## Troubleshooting

### "Another run is already in progress"

The fcntl lock on `.rivalclaw_run.lock` is held by another process.

1. Check if a run.py process is actually running: `ps aux | grep run.py`.
2. If the process is stuck/zombie, kill it: `kill <pid>`.
3. The lock releases automatically when the process exits. No need to delete
   the lock file.

### Protocol wallet drain (paper balance near $0)

The paper wallet can drain when many trades resolve as losses. The circuit breaker
halts trading when balance < `RIVALCLAW_RELOAD_THRESHOLD` ($100 default).

Fix: Reset the starting balance in the context table:
```sql
UPDATE context SET value='100000.00' WHERE chat_id='rivalclaw' AND key='starting_balance';
DELETE FROM context WHERE chat_id='rivalclaw' AND key='trading_status';
```

### Pending orders stuck

Orders in `pending` status that never received a Kalshi response are auto-cleaned
after 10 minutes by `reconcile_resting_orders()`. They are set to `cancelled`
with reason `stale_pending`.

If you need to manually clean them:
```sql
UPDATE live_orders SET status='cancelled', rejection_reason='manual_cleanup'
WHERE status='pending' AND submitted_at < datetime('now', '-10 minutes');
```

### Trades not executing

Check `risk_debug.log` for lines starting with `LIMIT`, `DEAD`, or `SMALL`:

- `LIMIT`: Asset or crypto exposure cap hit. Wait for positions to resolve or
  increase `RIVALCLAW_MAX_ASSET_PCT`.
- `DEAD`: Strategy score is 0.0 (killed by tournament). Check if the strategy
  is in `RIVALCLAW_DISABLED_STRATEGIES`.
- `SMALL`: Position sized below $0.10 after all multipliers. Usually means
  balance is too low or regime/score multipliers are compounding downward.

Also check:
- `RIVALCLAW_LIVE_KILL_SWITCH` is 0.
- Kalshi balance is above the floor.
- The series is in `RIVALCLAW_LIVE_SERIES`.
- The direction is not blocked (`RIVALCLAW_BLOCK_YES_TRADES`).

### Kalshi auth failures

If you see "401 Unauthorized" in logs:
1. Verify `KALSHI_API_KEY_ID` is set and correct.
2. Verify `KALSHI_PRIVATE_KEY_PATH` points to a valid PEM file.
3. Verify `KALSHI_API_ENV` matches your key (prod vs demo).
4. Check system clock -- RSA signature includes a millisecond timestamp.

## Daily Report Generation

`daily-update.sh` runs at 04:00 and produces `daily/YYYY-MM-DD.md` containing:

- Performance summary (balance, PnL, ROI, win rate).
- Latency breakdown (fetch, analyze, wallet, total cycle, signal-to-trade).
- Recent cycle metrics (last 10 cycles).
- Recent trades (last 5 closed).
- Tuner adjustments for the day.
- Market speed breakdown (fast/medium/slow tiers).

After generating the report, it:
1. Runs the Strategy Lab research cycle (diagnose, hypothesize, backtest).
2. Runs auto-promotion/demotion checks.
3. Generates the Strategy Lab daily report.
4. Commits daily/, strategies/, CHANGELOG.md, and strategy files.
5. Pushes to origin/main.

## Git Sync

**Hourly (commander):** At :27 past each hour, the external commander process
commits modified logs and state files, then pushes to both GitHub and GitLab.

**Daily (daily-update.sh):** At 04:00, commits reports and strategy files, pushes
to GitHub.
